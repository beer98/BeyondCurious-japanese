# 04 · 自进化复盘机制

> 系统能不能进化，关键在于"复盘 → 蒸馏 → 写入规则库 → 下次自动加载"这个闭环是否真正闭合。

## 一、三级复盘节奏

| 周期 | 时间 | 输入 | 输出 |
|---|---|---|---|
| 日复盘 | 当日盘后 30 min | 当日交易 + 当日观察 | `07-每日复盘归档/YYYY-MM-DD.md` 下半部 |
| 周复盘 | 周日 19:00–21:00 ET | 过去 5 个日复盘 | 更新 `08-知识沉淀/规则库.md` 候选区 |
| 月复盘 | 月末最后周日 | 4 周日复盘 + 持仓日志 | 把"经过 ≥3 次验证"的候选规则提升为正式规则；废弃失效规则 |

## 二、周复盘流程（90 min）

### Step 1 · 数据对齐（20 min）
- 把过去 5 天每个候选标的的实际涨跌、IV 变化、关键新闻列成一张表
- 区分：**已建仓 / 仅观察 / 主动跳过**三类

### Step 2 · 归因分析（30 min）
对每一笔已建仓和每一个跳过决策，回答：

- 判断对了还是错了？（事后 5 个交易日内验证）
- 是哪一层判断起了决定作用？（Layer 1 催化剂？Layer 4 情绪？）
- 这是**好判断 + 好结果** / **好判断 + 坏结果**（运气背） / **坏判断 + 好结果**（运气好，最危险） / **坏判断 + 坏结果**？

> 重点关注**坏判断 + 好结果**：这是未来爆雷的主因。

### Step 3 · 找模式（20 min）
连续 ≥3 次出现的模式才有意义。例如：
- "连续 3 次在 Reddit 散户狂热时建仓后回撤 >10%" → 写候选反规则
- "连续 3 次错过 Serenity 当周提到的票导致少赚 >5%" → 写候选正规则

### Step 4 · 更新规则库候选区（20 min）
把新模式写到 `08-知识沉淀/规则库.md` 的"候选区"。

格式：
```
### [候选] 反规则 · YYYY-MM-DD 提出
- 触发条件：
- 应执行动作：
- 已观察次数：1 / 3
- 关联日复盘：[2026-04-15](07-每日复盘归档/2026-04-15.md), ...
```

每次再次发生 → 计数 +1。达 3 次 → 月复盘时提升为正式规则。

## 三、月复盘流程（3 小时）

### Step 1 · 量化指标（30 min）
- 月度收益率 vs SPY / QQQ / SOXX
- 胜率（盈利交易数 / 总交易数）
- 平均 R:R
- 最大单笔亏损 / 月度最大回撤
- 期权策略命中率（区分 sell put / sell call / spread）
- **决策质量分均值**（来自日复盘第 13 项）—— 这个比 P&L 更重要

### Step 2 · 规则库提升 / 废弃（60 min）
- 候选规则达 3 次验证 → 提升为正式
- 正式规则连续 2 月命中率 < 50% → 标记"待观察"
- "待观察"再 1 月不达标 → 废弃，写入"墓地"区，**记录为什么当初以为它是规则**（防止再被发现一遍）

### Step 3 · 框架升级（90 min）
- 5 层漏斗的权重是否需要调整？（基于命中率回归）
- 是否需要新增一层？（例如发现"宏观流动性"是上游因子）
- 信息源是否需要替换？（哪些 X 账号 / subreddit 信噪比下降）
- 心法清单是否需要新增禁忌？

把所有调整以 git commit 形式记录到本仓库。

## 四、自进化的元规则

> 这些规则规约系统本身如何进化，不是规约交易。

1. **样本量 < 3 不立规则**——单次成功 / 失败大概率是噪音
2. **写规则必须可证伪**——"小心拥挤交易"不是规则；"X 上单条点赞 >10k 的票，盘前不追"是规则
3. **反规则比正规则优先**——避免大错比抓住每个机会重要
4. **规则之间冲突时，新的赢**——但要记录为什么旧的失效（市场结构变了？信号被套利了？）
5. **每个规则必须挂时间戳和首次提出的复盘链接**——便于追溯

## 五、给 AI 的指令

每次启动 Claude 之前，提示词里带上：

```
请先读取 08-知识沉淀/规则库.md 中的所有正式规则和反规则。
对今日候选清单的每一项，逐条检查是否符合 / 触犯规则。
触犯反规则的标的 → 自动剔除，理由写明引用了哪条反规则。
```

这就是"自进化"在 AI 端的实现：规则库就是系统的记忆。
